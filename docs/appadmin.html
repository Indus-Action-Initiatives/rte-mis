RTE MIS Product Documentation

App Administration

The app admin module is concerned with all features required for the administration of the application by the user who can be considered as super administrator of the application. It is the app administrator's responsibility to configure the application to suit the needs of the state government.

Multilingual Feature

This is a feature which is going to be implemented in the RTE-MIS portal. The RTE-MIS portal is going to be used in different states, and all states have their own language other than English. This is the reason why this multilingual feature is implemented in the RTE-MIS portal.

The basic language in the RTE-MIS portal is English, but there is an option through which the State can create their own translation of their state's language and can apply this to the portal.

  1. There should be a feature that can provide a facility for the user they upload a bulk translation. This may be in the form of an Excel sheet.
  2. Users can also have the option to add translations one by one.
  3. Only the State admin should be able to make translations and modifications to translations.
  4. There should be an option for the state admin to delete or modify any translation that comes on a need basis.

School UDISE Configuration

UDISE can be configured in two ways: first, by following the bulk upload process, and second, by having API integration. API integration is not followed in the MVP creation. But the process has been explained.

  1. In the bulk upload process, during the first year, since it will be time-consuming to onboard all the schools by asking all schools to fill in their data, the state will need the option to bulk upload school information through the backend. Process flow -
    1. A state or District person uploads a CSV or Excel file (a standard file with a predefined template for columns)
    2. States can upload for all districts, and districts can upload for their own district only.
    3. Once the file is uploaded, the system saves the school information, marks the account status as BULKUPLOAD and verifies it.
    4. In this process, there is the possibility that some schools that may be newly recognised are not available on the state's list. For the next years, to include the new UDISE code in the RTE portal, we need to provide a panel in the Block admins login where the Block-level authority can submit some basic details to the schools by offline validating the schools.
    5. School information that is added by the Block admin should be marked as INDIVIDUAL, and this needs to be verified by the higher authorities, that is District admin.
    6. DEOs can verify the new school's information by visiting the UDISE portal (from where schools will get the recognition and get assigned a UDISE code).
    7. Once the school's UDISE code is added to the RTE database either by BULKUPLOAD or by INDIVIDUAL upload process and the status is verified, then only the school is able to create an account to get registered for a particular campaigning year.
  2. Another way schools can be onboarded for the campaigning year is by having an API integration from the UDISE portal, through which UDISE configuration part can be skipped. API integration will be explained in the School account creation part (School management module Point 1).
  3. There should be a panel where the Block admin can perform the CRUD operations of existing School information. For example, on a requirement basis, BEOs can change the status of a school from non-minority to Minority, from non-aided to aided, and the school's active status to inactive and vice versa.
  4. In the case of API integration, integration of UDISE from the UDISE portal will happen onc, and then that particular UDISE will be disconnected from the API now if there is a change in the status of the school, that will be modified by the Block admin from the panel, which we will create in point 3.
  5. Fields that are required:
    1. UDISE code (Numeric only, mandatory, fixed to 11 digits, Non-primary key)
    2. School Name (alphabetic, mandatory)
    3. District Code (Number only, can be of a minimum of 3 digit max can be up to 5 or 6)
    4. Block ID/code (Number only, can be of a minimum of 3 digit max can be up to 5 or 6)
    5. U/ R (Alphabetic, one digit which keeps U for urban and R for rural)
    6. Minority (Y/N) (Boolen, 1 for Yes and 0 for No)
    7. Aided (Y/N) (Boolen, 1 for Yes and 0 for No)
    8. Active (Y/N) (Boolen, 1 for Yes and 0 for No)
Note:
  1. There should be a possibility that schools who do not have a recognition certificate and are not alloted UDISE code till the time the school registration/renewal process is going on. In this situation school admin can try to enter an invalid (or temporary or wrong) UDISE code to get the benefit of this scheme. There is also the possibility that the school will not get the recognition certificate for any reason. But with the wrong UDISE code, the school is registered and will get the admission of students, that is wrong. So to avoid this type of situation, this should be provided to the BEOs login, and there should be a declaration from the BEO that they have validated the schools. Also, next level of verification should be made by DEOs.
  2. BEOs will offline inform the schools that the school is added and verified now that the school can create an account for the campaign.
  3. This should be the responsibility of the District admin, that the records in Excel are already verified offline.
  4. This should not be verified from the portal itself, because we are not following the API integration through the UDISE portal. If the State follows the API integration, then there is no need for bulk or individual uploads. Also, the records will be directly verified from the External portal (that is UDISE portal) at the time of school account creation.
  5. For schools that are non-minority, non-aided, active and verified by DEO, only those schools can be eligible for account creation, and only the list of these schools will be displayed on the dropdown list to create an account.

State Location Configuration

Location should be configurable from state level, levels of the geographical locations will be fixed to eight, but the display name should be configurable as per states requirements or as the name of the area that is used by the State. There is a table where we are managing this state specific mapping information, that is which location name belongs to which table.

As the name of the levels has been defined and entered, then these names will be displayed to the portal. Now the end user can easily filter his/her living location from the list. Below are the level of location that is going to be followed:

  • State
  • District
  • Urban / Rural
  • Block
  • Nagariya Nikay (In case Urban)
  • Ward (In case Urban)
  • Gram Panchayat (In case of Rural)
  • Habitations (In case of Both)

There is a possibility that each state has their own names to call the level of locations, for this, there is a configurable section where, State admin can define the name of the locations which they are using for their state and this information will be stored on a table in the database.

Integration of SMS & Email Services

  1. The Email and SMS services are required in this portal, to validate the all admins authenticity. Email and SMS services are required to validate the user at the time of school account creation.
  2. The SMS services are the majorly used services in the RTE-MIS portal. The SMS services are going to be used to validate the schools at the time of account creation, further to get the verification status message.
  3. To recover the password an OTP verification will be made, and then the admin can update their password.
  4. Another use is in the student module, to send SMS to registered students to provide the confirmation of registration, and then from time to time the status of applications are going to be changed, they receive intimation through SMS.
  5. In the lottery process also, allotted applications will get the first intimation via the SMS to their registered mobile number.

Google Map Integration

Google map Integration is done for the school registration part which is mainly used to collect the latitude and longitude of the school's parent location. Purpose of collecting this information is to integrate the GIS location mapping with the parent's parent location to show them the list of schools to the parent for applying for the campaign. This should be limited to the 1 to 3 km radius as per the government rule.

Note: States can define their own distance criteria, so this should also be configurable.

Forgot Password Recovery

  1. There is a possibility that, user can forget their login password, in this situation the user can reset the password by visiting the reset password option in the login page.
  2. To reset the password, users enter the username and get an OTP on their registered mobile number and Email id, after validating the OTP, users are able to create a new password.
  3. Once the OTP is sent to registered mobile number and email id, then Ending of 4 numbers and crossed email id of registered mobile number and email id will display on reset page panel.
  4. OTP which is going to be sent on mobile number and Email ID should be the same.
  5. The Resend button will be displayed after 1 minute. Messages (OTP invalid or expired) should be displayed on the page clearly. Timer should also display.
  6. Flood control option should be there.
  7. In case a registered mobile number is not in use anymore, and the user is unable to get OTP on registered mobile number, then he/she needs to visit their higher authority to get their registered mobile number updated.
  8. Dedicated higher authority can update the registered mobile number, this update also requires an OTP validation. Due to this, the user needs to be physically present with their dedicated higher authority.
  9. Once the new mobile number is registered, then the user will be able to reset their password.
  10. The user's level should be like School should go to Block admin, Block admin request to District admin, District admin request to the State admin, and in case of State admin app admin or IT admin will directly update the details.
Notes:
  1. User name should be UDISE code for schools and email id for all govt admins and roles.
  2. Incase of user forgot their registered email id which is going to be used as a username, then the user can request higher authority to change or to recover the username. Then the IT admin will directly provide or update the email id which is going to be used as username.
  3. Passwords should be hashed.

School Mapping

  1. School mapping (map a school to nearby neighborhood areas) can be done in two ways, either by manual mapping or by the GIS mapping. For the mapping process a block admin should follow the distance rules that will be defined by the State.
  2. In the case of a manual mapping portal will not capture the distance information but in the case of GIS mapping portal needs to collect the distance information, through this portal will calculate the distance between school and students residence, and on the basis of distance saved, list of schools will display to the parent to select the school. In the MVP we are not covering the GIS option, this will be customisable as per State’s requirement.
  3. Block admin will be the person who will be responsible for doing mapping of a school to nearby habitation.
  4. Once a school is verified from the block level (first level approval), then immediately they can do the mapping of approved schools to nearby neighborhood areas as per State govt rules.
  5. Mapping should be renewed every year, this should be mandatory for block admins to re check the mapping and if any changes require then after doing those changes Block admin should submit the changes.
  6. This should also be the responsibility of the district admin to verify those changes and approve them on the portal.
  7. Once the mapping will get approved then that school will be enabled to take admission for the current campaign cycle.
Notes:
  1. We should have a section to identify the duration for school mapping in the campaign creation page. In this we need to define the Start date and end date selection option to perform mapping work.
  2. There should be one level approval option, Block admins will add the habitations to the schools and District admins will verify this while making approval of school registration or renewal.
  3. Mapping will be done after the first level of verification which will be done by the block admins.
  4. Whenever the renewal of mapping will happen, at that time existing mapping records should display, and block admin can be able to make necessary changes as per requirement.
  5. If Mapping is not approved for the particular campaigning cycle, then that particular school should not display for the student application.
  6. There should be a view panel to the block admin to see the records of the school, through which block admins can see the list of schools with current status. Same should be available to the District admin also, to see and to make verification of schools.
