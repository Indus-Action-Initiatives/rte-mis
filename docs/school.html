RTE MIS Product Documentation

School Management

In School Modules school administrations enroll themselves for specific academic cycles. To participate in the RTE scheme a private school must have an account in the MIS portal. Once the account is created, the school can register itself. Further from the preceding years, schools need to modify their existing information.

In this module school administrators provide schools information which are required for participating in the scheme. Information like - location information, recognition information, seat information, entry class information, fees information, CWSN availability information, bank account details etc.

Once the registration process is completed, the government authority performs the verification and makes necessary mapping operations to the verified school.

School Account Creation

At the time of school account creation, there should be two ways one is by fetching the records from the RTE database or by fetching the records using the API integration of UDISE portal.

For collecting the school information from RTE database schools should follow the below process:

  1. School person visits the RTE portal and clicks on the school account creation tab and enters the UDISE code.
  2. System checks for the validity of entered UISE code:
    1. In first step checks entered Udise code is belongs to an existing school in RTE portal database, and its status is not BULKUPLOAD or INDIVIDUAL (this may be approved, reject or any other) If yes, then throws an error that an account for school with entered UDISE is already exists, and notify that user can reset their credentials using forgot password tab.
    2. If status is BULKUPLOAD or INDIVIDUAL then, the school moves to the next step and enters other information which are:
      • Mobile Number
      • Email ID
      • District
      • Block
      • U/R
  3. After entering the above information, validate the Mobile number and Email ID through the OTP validations.
  4. There will be a check box where the school admin will check on declaration and accept that information that was submitted by them is valid.
  5. Once the validation is done then the school can submit the details and an account will be created.
  6. Once the account is created then the school will be able to do the campaign registration.
    7. Note:

    Till the campaign registration is not approved, school should be in block status.

School Registration - Basic + Address + Seat Details + Entry Class

New school registration has been splitted into two sections one which is mandatory and linked to the student management module and the other is related to the reimbursement module.

In this section we are going to perform the registration for which section which is required for the student module.

  1. Once email and mobile number are verified, then school moves to the next step for the registration process and proceeds with the below process. If the email address and/ or mobile number is not verified then the registration process will stop or rollback with the notification that “Without verification of email and mobile number school can not part in the registration process”.
  2. Once UDISE code, email and mobile number has been verified then the school can register for a new academic year. Below is the list of required field information that need to be collected in RTE portal:
    1. Basic information:
      • AC_Year (Read only)
      • Recogination_Year (text field)
      • Udise code (Read only)
      • School name (Read only)
      • Full name of school administrator (Alphabetic, mandatory)
      • Mobile number (Read only)
      • Email (Read only)
      • School website (Alphabetic, non mandatory)
      • Status of aid (Dropdown unaided, aided) (Read only)
      • Status of Minority (Dropdown Minority, Non Minority) (Read only)
      • School recognition certificate number (alphanumeric , mandatory)
    2. Address details:
      • District (Read only)
      • Block (Read only)
      • Urban or Rural (Read only)
      • Nagariya Nikaye (display on selection of Urban) (dropdown)
      • Ward (display on selection of Urban) (dropdown)
      • Gram panchayat (Display on select of Rural) (dropdown)
      • Habitation (dropdown)
      • Full address of the school including PIN Code (text field, mandatory)
      • Google Maps location (collect lat, lng of schools parent location) (mandatory)
    3. School type declaration:
      • Board type (Local board, CBSE, ICSC) (configurable, Mandatory, dropdown)
      • Medium of instruction (Hindi, English, Local language) (configurable, Mandatory, dropdown)
      • Level of education (Primary, Upper Primary, High, Higher Sec.) (configurable, Mandatory, dropdown)
      • School Type (Co-educational/Mixed, Boys, Girls) (configurable, Mandatory, dropdown)
      • From class (dropdown)
      • To class (dropdown)
    4. School type declaration:
      • Board type (Local board, CBSE, ICSC) (configurable, Mandatory, dropdown)
      • Medium of instruction (Hindi, English, Local language) (configurable, Mandatory, dropdown)
      • Level of education (Primary, Upper Primary, High, Higher Sec.) (configurable, Mandatory, dropdown)
      • School Type (Co-educational/Mixed, Boys, Girls) (configurable, Mandatory, dropdown)
      • From class (dropdown)
      • To class (dropdown)
    5. Entry class and seat declaration:
      • Class (Which is entry class for school)
      • Total seat
      • Medium
      • Promoted std (Mandatory for class -1 in case of dual entry class)
    6. Check box of Declaration:
      • At the end of the form school admin accepts the declaration that all information they filled are correct and incase of mis declaration govt is authorized to take action against them.

    Once the above information is filled and the checkbox is checked then the school admin will submit the form. After the successful submission, the school admin takes the printout of registered information and with necessary documents submit it to the dedicated block education officer.

School Registration Renewal

  1. School registration renewal process will be applicable to the schools when a school already exists in the portal and for a previous campaigning year that school has already participated.
  2. In this situation a school logs in into the portal and then selects Renewal option from the menu. After this schools need to make necessary changes and click on the renewal button and do the modifications and submit it to the respective block officer.
  3. Once the changes are made by the schools, that will reflect on their profile after the verification process. As there are two levels of verification process will follow, so the changes will reflect after the second level of verification.
  4. After the modification made by the school the status of schools should move to the “Registered” status. And after the verification completed then status should change to the verified and then approved.
  5. There should be a section in the Dashboard where the verification status will be displayed to school.

Note:

  1. Once a school has pressed the renewal button and form is submitted, then till the application status is not moved to verified status or rejected status till then school can do the modifications on the form if any further modification require.
  2. If any situation arises, a school has opened the form to make further changes and at the same time block has opened his form to make verification. In that situation, from the backend, code should be checked for the current status of the form and then only apply the changes in the form.
  3. No application id will be generated for the school registration or renewal process. Only one time registration will allow and then modification can be made, and renewal will also be made on the same information. Composite keys can be generated. This composite key will not be visible to school.
  4. List of things that are not modifiable at the time of renewal process:
    1. UDISE code
    2. School name
    3. Type of School (aided / unaided)
    4. Type of Minority
    5. District
    6. Block
    7. Urban Rural option
    8. Entry class (At the time of registration this should be modifiable after that this should not be modifiable)

School Registration Print

  1. From this section, schools should be able to take printouts of their registration information’s printout at any point of time whenever they require.
  2. To take printout of registered information by log in to their account and by click on the profile menu and click on print button.

School Registration Auto Renewal

There is a possibility that some schools have not done the renewal of their registration which they submitted in previous academic year, and duration for renewal has been closed for school admins.

  1. In this situation, there will be a panel available for the block admin to perform the verification process.
  2. In that panel block admin will have a view page where they can see the list of schools with the current status of schools.
  3. Block admin select the school who has not performed the renewal process, and cross verify the previous years information and mark as verified.
  4. Once the school is marked as verified then that school will be available for DEOs to perform the final approval process.
  5. Block admins need to check each school individually and mark as verified.
  6. This panel will be open to the Block admin when the School registration or renewal window will close for the school. This should not be accessible to DEO.
  7. Create Campaign for this section that should be manageable by the State.

School Registration Auto Verification

  1. There is a possibility that the Block admin has not done the verification of the schools new registration or renewal and time period has expired for the BEO to verify.
  2. There is also a possibility at the district level that the District admin has also not done the verification of schools for the second level approval.
  3. In this situation there will be a panel where the District admin can do the verification of all pending schools on a single click.
  4. District admin will have a view panel where they can see the list of all pending schools. There should be filtration options for district authorities through which they can do verification in two different ways.
  5. First is approval of those schools which are pending from the Block admins to get approved, these applications need two levels of verification. For this district admin first filter the list of schools which are pending from the BEO end and perform the verification process and from the backend two levels of verification process happens for all the schools which are shortlisted are marked as verified.
  6. Second one is, the schools which are approved from the BEO end but are pending from the DEO end. In this situation on the same view panel DEO will perform the filtration process and get the list of those schools which are pending from the DEO end. After getting the list of schools DEO approves the schools and from the backend single verification code will run.
  7. In both cases there will be a bulk check box which will be by default selected for them, and DEO can perform the uncheck process for those schools to whom they don't want to verify.
    Don’t want to verify means -
    1. Schools who are not registered or not applicable, don’t have enough documents.
    2. Schools who moved to minority or aided.
    3. Schools who get closed or not get renewed COR (Certificate of recognition).
  8. There will be another panel which is for DEOs to search for a particular school using Udise code and DEO can individually reject or approve that particular school. This process is to reduce the search process of schools which is going to be performed on point g that needs to get unchecked from the list.
  9. This panel will be open to the District admin when the Block and District admins verification window will close for them.
  10. There will be an option in the Campaign creation panel, where the state admin can define start date and enddate.

School Registration Verification

State can follow one level of verification or two level of verification, process are as follow:

  1. IF State chooses the two levels of verification, in that situation a block level officer completes the verification process (offline), and then officer must change the status of the school from “Applied” to “Verified” with some remarks.
  2. If they find that modification is needed in the form then officers change the status to “Send Back to School” and ask the school to make necessary changes and resubmit their application.
  3. If during the verification process it was found that the school is not eligible for participating in the campaign, the officer will write reason on remarks and will change the status of the school to “Rejected”. Once the school moves to “Rejected” status, after this school should not be able to make modifications on application.
  4. The district level officer should be able to see the list of schools marked as “Verified” by the block level officer. If the district officer is satisfied the officer will change the status of the school to “Approved”.
  5. Once the District level officer is marked as” Approved”, at that moment the school admin will get the access to all necessary modules for which school is eligible for.
  6. In other way, if the State takes a decision for one level verification, then it should be possible to verify both level verification by a single user.
    7. Notes:
    1. Level of school registration verification should be configurable, either two levels (BEO →DEO) or one (BEO or DEO). Need to set up the access rights on the basis of approval.
    2. Schools will get SMS notifications of status getting changed. This should be configurable at the State level, whether they want to have SMS notification or not.
    3. This should be the decision of the state that they want to open the registration / modification / renewal process and verification process together or separately. This can be managed by a campaign creation page ( no configurable pages are required).
    4. If this is not opening together in that situation, there should be a school registration modification page required at the level of BEO, from where BEOs can make necessary changes and mark applications directly to the “Approved” status.
    5. During the verification duration BEO can change all information about the school. Only those fields should not be editable which are not editable by schools also like - Mobile number, email id, district, block, U / R, Udise code, school name etc.
    6. If State follows two levels of verification, in that situation if DEOs want to get some modification on school registration then school can reach to BEO and after getting necessary changes school can get their registration to be approved.
    7. Once the verification window will get closed then there should be a panel for DEOs from where they can mark all schools as verified and approved status by bulk check box option.
    8. If BEO marks school to “Rejected”, after that registration process is locked for school and after that if school is found eligible then, BEOs can move the Status from “Rejected” to “Send Back to School” status and after that school will be eligible to make necessary changes and re submit the application.
    9. Log should be maintained for each and every time status will get changed.

RTE Seat Declaration Logic

There are certain cases for declaring the entry class or classes. This should be the configurable section for the State administration. In all cases Class -1 should be mandatory.

  1. Dual entry class - In dual entry classes, class -1 will be in the form of a mandatory class, and another will be either Nursery (LKG) or KG-1 (UKG).
    • This should not be mandatory for all schools to have dual entry class, schools can have either only the class - 1 as entry class or with the class - 1 can have any one from the pre- primary classes.
    • While School is going for dual entry classes then certain backend calculations should be performed to make the division of % correctly.
    • In case of dual Classes, while entering the total seats information for class - 1 then below calculation should performed - (Total seat of class 1) - (Total seat of pre- primary) as Remaining Total seat for class - 1, and then State specified percentage will be taken to declare as Total RTE seat for class - 1.
    • If after performing above calculation, class -1 seat comes 0 or less then 0, in that case declare the RTE seat for class - 1 to 0 (Zero). Or remove the class -1 from entry class.
    • And in case the school wants to remove pre- primary from entry class then this is up to the school, they can make modifications on their registered information via profile update section.
  2. Single Entry Class - In single entry class, only class -1 will be declared as the entry class for all schools.
